永続化の正は Neon PostgreSQL だけ

Git にはコードと雛形、編集中はブラウザメモリ。localStorage / JSON は v1 では使わない

出典: eu-co2-simulator-grill-decisions.md §17（2026-06-11 グリル合意）

PostgreSQL とは？ なぜ今回これを使う？

PostgreSQL（ポストグレスキューエル）は、データを行と列の表（テーブル）で整理して保存・検索するデータベース（DB）ソフトウェアです。Excel の表に近いイメージですが、アプリ専用の金庫のように、大量データの一覧・更新・削除を安全に扱えます。

PostgreSQL

表形式のリレーショナル DB（表同士を関連でつなげる方式）。SQL という言語で「保存して」「条件で探して」と指示する。

Neon

PostgreSQL をクラウド上で借りるサービス。Vercel 上のアプリが DATABASE_URL 経由で接続する。

このツールで PostgreSQL を「永続化の正」にした理由

  • Vercel 公開 — ブラウザ内（localStorage）だけでは端末を替えても同じデータが見えにくい
  • 名前付きシナリオを複数 — 一覧・上書き・別名保存・削除を表で管理しやすい
  • 保存時に 3 年分の計算結果 — サーバー側 DB へまとめて書き込める
  • 機密データ — 台数・CO₂ は Git に入れず DB に預ける（Basic 認証付き）

JSON ファイルはデータの書き方（形式）、PostgreSQL はデータの保管・検索の仕組み。旧合意の localStorage / JSON は v1 では使わず、ユーザーが「保存」した内容だけを Neon に残します。

Git リポジトリ

デプロイに含まれる静的資産。ユーザーが保存した機密データは入れない。

  • db/schema.sql
  • lib/defaults.ts（雛形の正）
  • lib/constants.ts
  • db/seed.sql（dev のみ）

クラウド上のデータベース。名前付きシナリオと計算結果が残る。

  • scenarios
  • portfolio_rows
  • regulation_params
  • weight_coefficients
  • scenario_results ×3年

編集中だけ。リロードで消える。明示「保存」まで DB に書かない。

旧合意と現行合意 — 保存形式の違い

このツールの「データをどこに残すか」は、2026-06-11 のグリル（§17）で方針が更新されました。旧合意（2026-05）は撤回済みですが、設計の背景を理解するために両方を並べて説明します。

旧合意（2026-05）— 撤回

個人 PC のブラウザ内 + JSON ファイル

  • 実行: 自分の PC で npm run dev のみ
  • 永続化: localStorage 自動保存
  • 入出力: JSON エクスポート／インポート
  • クラウド DB なし・Vercel 公開なし
現行合意（2026-06 §17）

Neon PostgreSQL（クラウド DB）のみ

  • 実行: Vercel + Neon で公開
  • 永続化: PostgreSQL のみ（明示「保存」）
  • localStorage / JSON は v1 対象外
  • Basic 認証・名前付きシナリオ複数

JSON と PostgreSQL — このプロジェクトでの意味

JSON（旧合意で想定）

データの書き方（テキスト形式）。1 ファイルに画面状態をまとめ、コピーして別 PC へ渡す用途向き。

旧: エクスポート／インポート用　／　現行: v1 未実装

PostgreSQL（現行の正）

データの保管・検索の仕組み。複数シナリオの一覧、保存時の 3 年分計算結果など、表で管理。

現行: 永続化はここだけ

なぜ方針が変わったか

  • 実行環境が「個人 PC のみ」から「Vercel 公開」へ → ブラウザ内（localStorage）だけでは端末を替えたときにデータが追従しにくい
  • 名前付きシナリオを複数残したい → DB の方が一覧・上書き・別名保存・削除を整理しやすい
  • 保存時の計算結果（3 年分）も残す → サーバー側 DB に UPSERT する設計が自然
  • 機密データをBasic 認証付きクラウドで扱う前提 → 共有 DB プール + 簡易保護

以下は現行合意（§17）の詳細です。

4つの「置き場所」

Git リポジトリ（ギット・レポジトリ）は、ソースコードを保管する場所です。PostgreSQL（ポストグレスキューエル）は、表形式でデータを永続保存するデータベースです。Neon は、その PostgreSQL をクラウド上で使えるサービス名です。

置き場所 役割 中身の例
Gitコード・設計図schema.sql, defaults.ts, constants.ts
Git（dev のみ）DB リセット用サンプルseed.sql
Neon DBユーザー保存データ名前付きシナリオ・計算結果
ブラウザメモリ編集中の一時状態SimulatorState
Git に入れない機密台数・CO₂・保存済みシナリオ

たとえ話：引き出しと金庫

Git は設計図と型紙が入った引き出し（誰が見ても同じ）。Neon はあなたが書き込んだノートが入る金庫（保存ボタンを押したときだけ中身が増える）。編集中のメモは机の上（ブラウザメモリ）に置いているイメージです。

DB の中身：スナップショット型

スナップショット型（スナップショット＝その瞬間の写真）とは、「保存ボタンを押したときの画面状態を一式として丸ごと残す」設計です。中心は scenarios テーブル1行＝名前付きシナリオ1件です。

scenarios（シナリオの入れ物）
├─ portfolio_rows … Pane1 車種行（2025表 / 2026表）
├─ regulation_params … Pane2 規制（2025/2027/2030）
├─ weight_coefficients … Pane2 詳細 重量補正 a, M₀
├─ scenario_results … Pane3 計算結果（保存時3年分）
└─ ai_analyses … v1 未使用

画面（Pane）と DB テーブルの対応

Pane 1
portfolio_rows

2025 実績表 / 2026 予測表を base_year で区別。車種名・動力方式・台数・WLTP・重量。

Pane 2
regulation_params + weight_coefficients

規制パラメータは 3 年分。重量補正係数はシナリオと 1:1。

Pane 3
scenario_results

保存時に 2025 / 2027 / 2030 をサーバー計算。達成率・罰金など。

Pane 4
（DB に保存しない）

AI 文案はボタン押下のたびに再生成。ai_analyses テーブルは v1 未使用。

Basic 認証（共有 DB）

defaults.ts と seed.sql の関係

誤解

2 ファイルは別々のデータ源で、どちらか一方を直せばよい。

正解

lib/defaults.ts が正。seed.sql は同じサンプルを SQL で書いた開発用コピー。更新時は defaults を先に直し、seed を手動で揃える。本番 Neon には seed を流さない。

例：「コンパクト（ICE/HEV）12500 台 118 g/km」は defaults.ts と seed.sql の両方に同じ数字で書かれています。これが二重化（同じ内容が 2 か所にある状態）です。

現行 v1 で使わないもの（旧合意にあった機能）

以下は2026-05 旧合意では採用予定だったが、2026-06 現行合意（§17）で v1 対象外としたものです。

  • localStorage 自動保存 — 旧: 編集内容をブラウザ内に自動記録。現行: 使わず、明示「保存」で DB へ。
  • JSON エクスポート／インポート — 旧: ファイルでバックアップ・PC 間移行。現行: v1 未実装（PostgreSQL が正）。
  • ai_analyses への INSERT — AI 文案は都度再生成
  • ユーザー別ログイン — v1 は共有 DB プール

Git に入れてはいけない環境変数（.env）

  • DATABASE_URL — Neon 接続文字列
  • BASIC_AUTH_* — 認証用（Phase 2）
  • LLM API キー — Pane 4 任意送信
